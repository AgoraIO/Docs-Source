The <Vg k="AI_FRAMEWORK" /> distinguishes between different messages using message types, each of which provides relevant functionalities. For example:

- A status command type message requires a response and has a specific message type assigned to it.

- A message containing an image buffer has a corresponding message type that provides APIs for getting or setting the frame data.

- A message containing an audio buffer or "pcm frame" has a dedicated message type with APIs for accessing or modifying the frame data.

The following is an overview of <Vg k="AI_FRAMEWORK" /> messages.

```
┌── has response
│   └── command
│       ├── TEN framework built-in command
│       │    => message names start with `rte::`
│       └── Non-TEN framework built-in command
│            => message names do not start with `rte::`
└── no response
  ├── data
  │   ├── TEN framework built-in data
  │   │    => message names start with `rte::`
  │   └── Non-TEN framework built-in data
  │        => message names do not start with `rte::`
  ├── image_frame
  │   ├── TEN framework built-in image_frame
  │   │    => message names start with `rte::`
  │   └── Non-TEN framework built-in image_frame
  │        => message names do not start with `rte::`
  └── pcm_frame
    ├── TEN framework built-in pcm_frame
    │    => message names start with `rte::`
    └── Non-TEN framework built-in pcm_frame
       => message names do not start with `rte::`
```

Message names are used to differentiate the purpose of messages within the same message type.

## Message Type

The <Vg k="AI_FRAMEWORK" /> supports the following message types:

1. command
1. data
1. image frame
1. pcm frame

The difference between commands and non-commands is that commands have a response, referred to as a status command in the <Vg k="AI_FRAMEWORK" />, while non-commands do not have a response.

The corresponding extension message callbacks are:

1. `OnCmd`
1. `OnData`
1. `OnImageFrame`
1. `OnPcmFrame`

Although there are currently only four message types in the <Vg k="AI_FRAMEWORK" />, it is possible that more types may be added in the future. Considering this, a possible design was to merge all the extension message callbacks into a single `OnMsg` callback to avoid the issue of adding new callbacks for each new message type. However, this would inconvenience the developers, as it means conditionally handling different message types within the same `OnMsg` function.

```cpp PseudoCode
OnMsg(msg) {
  switch (msg->type) {
  case command || connect || timer || ...:
    OnCmd(msg);
    break;
  case data:
    OnData(msg);
    break;
  case image_frame:
    OnImageFrame(msg);
    break;
  case pcm_frame:
    OnPcmFrame(msg);
    break;
  }
}
```

Furthermore, in the TEN graph there is a distinction between `cmd_in`, `cmd_out`, `data_in`, `data_out`, and so on. Therefore, following a unified approach would still require differentiation of message types.

<Admonition>
If no message type is specified, the default type is `cmd`.
</Admonition>

## Message Name

The <Vg k="AI_FRAMEWORK" /> uses message names to identify the message purpose under the same message type. The extension determines what actions to take based on the differentiation of message names.

The naming convention for message names is as follows:

- The first character can only be a-z, A-Z, or _.
- Other characters can be a-z, A-Z, 0-9, or _.

## Creating messages

Non-<Vg k="AI_FRAMEWORK" /> built-in command:

```json
{
  "rte": {
    "type": "cmd",           // required
    "name": "hello_world"    // required
  }
}
```

<Vg k="AI_FRAMEWORK" /> built-in command:

```json
{
  "rte": {
    "type": "cmd",           // required
    "name": "rte::connect"   // required
  }
}
```

Data:

```json
{
  "rte": {
    "type": "data",          // required
    "name": "foo"            // optional
  }
}
```

Image Frame:

```json
{
  "rte": {
    "type": "image_frame",   // required
    "name": "foo"            // optional
  }
}
```

PCM Frame:

```json
{
  "rte": {
    "type": "pcm_frame",    // required
    "name": "foo"           // optional
  }
}
```

## TEN Runtime message name optimization

Within the TEN runtime, each message is recorded with two fields:

- Message type: Enum
- Message name: String

Since message names are in string format, there is a certain performance overhead when comparing strings. Therefore, when the TEN runtime encounters certain specific message names, it can use a message type to represent that message. For example, when it sees a message name like rte::connect, the TEN runtime can optimize it from:

- Message type: cmd (required)
- Message name: rte::connect (required)

to:

- Message type: connect (required)
- Message name: rte::connect // optional

This optimization is only applied to message names recognized by the <Vg k="AI_FRAMEWORK" />, and therefore, it is only possible for <Vg k="AI_FRAMEWORK" /> built-in message names. You cannot perform this optimization on your own by adding  new message types. Since built-in messages are limited, the newly added message types are also limited, and align with the enum type of message types.

In summary, message types other than `cmd`, `data`, `image_frame`, and `pcm_frame` are generated through this optimization, and these additional message types correspond to a special message name, which is always a TEN built-in message.

This optimization is not only applicable to commands but also to other message types. For example:

- Message type: image_frame (required)
- Message name: rte::empty_image_frame (required)

can be optimized to:

- Message type: image_frame_empty (required)
- Message name: rte::empty_image_frame (optional)

Since the <Vg k="AI_FRAMEWORK" /> modifies the type field of the message, you can also see and use this optimization. Leverage this feature to speed up message analysis. You can use two methods to identify the message:

1. Using the message name

    ```cpp
    if (message_name == "rte::timer")
    ```

1. Using the message type

    ```cpp
    if (message_type == MSG_TYPE_TIMER)
    ```

## Creating optimized messages

<Vg k="AI_FRAMEWORK" /> built-in command:

```json
{
  "rte": {
    "type": "cmd",          // mandatory
    "name": "rte::connect"  // mandatory
  }
}
```

or

```json
{
  "rte": {
    "type": "connect"       // mandatory
  }
}
```

## Specifying message type or message name

If it is not possible to determine the message type or message name from the context, it is necessary to specify the message type or message name.

For example:

- Message from JSON

    When the message type or message name is unknown, it is necessary to specify the message type and message name in the JSON.

- Command from JSON

    When it is known that the message is a command but the command name is unknown, it is necessary to specify the message name in the JSON.

- Connect command from JSON

    When it is known that the message is a `connect` command, the message name can only be rte::connect. Therefore, it is not necessary to specify the message type or message name in the JSON.
